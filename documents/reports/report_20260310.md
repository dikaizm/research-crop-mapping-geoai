### 4. `XAmzContentSHA256Mismatch` ‚Äî Root Cause Identified

**File changed**: `mlflow-research/Dockerfile`
**File created**: `notebooks/boto3_s3_test.ipynb`

#### Background

The artifact upload chain fails with:
```
boto3.exceptions.S3UploadFailedError: ... XAmzContentSHA256Mismatch when calling the PutObject operation
```

#### Root Cause (Confirmed via Direct Testing)

**Not** a payload signing issue. The actual cause: **boto3 ‚â•1.34 enables `request_checksum_calculation=when_supported` by default**, which adds CRC32 checksum headers (`x-amz-checksum-crc32`) to every `PutObject` request. MinIO rejects these headers with `XAmzContentSHA256Mismatch`.

The previous fix (`payload_signing_enabled = true` in `~/.aws/config`) addressed a different older issue (streaming SHA256 placeholder) and does NOT fix the boto3 ‚â•1.34 checksum behavior.

#### Confirmed Fix

```python
# Python client
from botocore.config import Config
s3 = boto3.client("s3", ..., config=Config(
    request_checksum_calculation="when_required",
    response_checksum_validation="when_required",
))
```

```dockerfile
# Dockerfile (server-side boto3 for proxied uploads)
ENV AWS_REQUEST_CHECKSUM_CALCULATION=when_required \
    AWS_RESPONSE_CHECKSUM_VALIDATION=when_required
```

Verified: direct `upload_file` to `s3://research-geoai/mlflow/artifacts/test-boto3-direct/fixed.txt` succeeds with the fix; fails without it.

### 8. `notebooks/04_feature_analysis_v2.ipynb` ‚Äî ASTFS-Inspired Per-Crop Band Selection

**File created**: `notebooks/04_feature_analysis_v2.ipynb`

Created a new feature analysis notebook implementing an ASTFS-inspired (Yin et al. 2020) per-crop band selection approach, replacing the global joint-score approach in v1.

#### Motivation

In v1, Stage 1 ranks bands by their global mean GSI across all crops, and Stage 2 trains a multiclass oracle (11 classes, metric = mIoU). This collapses per-crop spectral requirements into a single global ranking, potentially masking features that are highly discriminative for rare or spectrally similar crops. The ASTFS method instead computes a **per-crop SIglobal** and runs a **binary forward selection** (crop_i vs. rest) for each crop independently.

#### Key Differences vs v1

| Aspect | v1 (global) | v2 (ASTFS-inspired per-crop) |
|---|---|---|
| Stage 1 ranking | Global mean GSI + RF joint score | Per-crop SIglobal (`gsi_df[crop_id]`) |
| Stage 2 oracle | Multiclass U-Net (11 classes), metric = mIoU | Binary U-Net (2 classes), metric = IoU(class 1) |
| Output | Single K* band list (global) | Per-crop K*_i + union ‚Üí Stage 3 Exp C |
| Candidates | One ranked list of 25 | One ranked list of 20 per crop (10 lists total) |

**Reference**: Yin et al. (2020). "Automatic Spectro-Temporal Feature Selection for Crop Mapping." Remote Sensing 12(1):162. https://doi.org/10.3390/rs12010162

#### Notebook Structure

| Section | Key implementation |
|---|---|
| Config | Same paths/params as v1; added `TOP_K_PER_CROP=20`, `CDL_CLASS_NAMES` dict |
| Data loading | Identical to v1 (S2 stack + CDL + pixel sampling at `SAMPLE_FRACTION=0.05`) |
| **Stage 1** | `gsi_df[crop_id].sort_values()` per crop ‚Üí `candidates_per_crop[crop_id]`; SIglobal heatmap (crops √ó dates) |
| **Stage 2** | `RasterPatchDataset` upgraded with `remap_lut` and `target_class_id` params; `train_eval_binary()` trains binary U-Net; per-crop loop logged under `crop_{id}_` MLflow prefix |
| **Stage 2.3** | Per-crop results table {Crop \| Key Dates \| Key Bands \| K*_i \| Final IoU(class 1)}; union computation; per-crop IoU progression subplots |
| Stage 3 note | `stage3_exp_c` = union of per-crop selections ‚Üí Exp C input for `05_train_segmentation.ipynb` |

#### Design Decisions

**`RasterPatchDataset` upgrades:**
- `remap_lut=None` ‚Äî pass a custom binary LUT `{crop_id‚Üí1, else‚Üí0}` per crop oracle; defaults to global multiclass REMAP_LUT for backward compatibility
- `target_class_id=None` ‚Äî if set, only include patches with ‚â•1 pixel of the target CDL class; ensures binary oracle trains on patches that have positive examples (critical for rare crops such as Grapes at 1.3%)

**`train_eval_binary()` vs `train_eval()` (v1):**
- `num_classes=2` (binary)
- `CrossEntropyLoss()` without `ignore_index=0` ‚Äî in binary mode, class 0 (everything that is not the target crop) is informative; suppressing it would remove most of the supervision signal
- Metric: `compute_iou_class1()` = IoU of class 1 (the target crop) instead of mean IoU over all foreground classes

**Per-crop SIglobal rationale:**
`gsi_df = bs.calculate_gsi(df, CLASS_COL)` returns shape `(n_bands, n_classes)`. Column `gsi_df[crop_id]` is the SI between crop_id and all other classes, averaged over all pairings ‚Äî exactly SIglobal(b) from ASTFS Eq. 2. Using the per-crop column instead of the global mean (`gsi_df.mean(axis=1)`) means that, e.g., a band that is excellent for distinguishing Rice from everything else but mediocre for Grapes will rank near the top for Rice but not for Grapes ‚Äî a more targeted selection.

**Stage 3 union:**
The union of all per-crop selected bands (in first-occurrence order across crops) becomes `stage3_exp_c`. This is expected to produce a set of 15‚Äì30 bands that is both compact (vs. 44 for Exp B) and multi-crop optimal (vs. 4 for the v1 K*=4 under-selected result).

#### Key Design Decisions

**`scripts/config.py` as single source of truth**: All hyperparameters (`PATCH_SIZE`, `BATCH_SIZE`, `MAX_EPOCHS`, `S2_DELTA`, etc.) and paths are defined once here. Both `feature_analysis.py` and `train_segmentation.py` import from `config.py` ‚Äî no duplicated constants across scripts or notebooks.

**Exp C band loading (`train_segmentation.py`)**: Reads `stage3_exp_c_bands.txt` (one `BAND_MMDD` per line) and maps each entry to a flat local index via `mmdd_to_date[mmdd]` ‚Üí `local_band_to_idx[f"{band}_{local_date}"]`. This is year-agnostic: the same MMDD offset maps correctly to 2022, 2023, or 2024 files as long as they share the same acquisition calendar.

**Skip logic**: Each script checks whether its primary output already exists (`stage3_exp_c_bands.txt` for feature analysis; `best_model.pth` per experiment for training) and skips unless `--force` is passed. This makes partial reruns safe.

**Background execution**: `run.sh` uses `nohup ... > >(tee $LOG_FILE) 2>&1 &` so output is visible in terminal AND saved to a timestamped log. PID is saved to `logs/pipeline_YYYYMMDD_HHMMSS.pid` for easy `kill`.
