#### Key Client-Side Requirements Discovered

- **`MLFLOW_DISABLE_TELEMETRY=true`** must be set **before** `import mlflow` ‚ÄĒ MLflow 3.x initiates a background connection to `config.mlflow-telemetry.io` during `set_experiment()` that blocks indefinitely when the host is unreachable (common in restricted network environments).
- **HTTPS required**: Cloudflare enforces a 308 redirect from `http://` ‚Üí `https://`; the MLflow Python client does NOT follow redirects (hangs indefinitely). Tracking URI must be `https://mlflow-geoai.stelarea.com`.

### 3. MLflow Server Configuration Issues Diagnosed and Fixed

A series of issues were identified and resolved across multiple deployment cycles.

#### Issue 1 ‚ÄĒ `NoCredentialsError` on `log_artifact`

| | |
|---|---|
| **Symptom** | `botocore.exceptions.NoCredentialsError` on artifact upload |
| **Cause** | Server running `--default-artifact-root s3://...` ‚ÄĒ artifact URI returned to client was `s3://research-geoai/...`; client tried direct S3 upload with no credentials |
| **Fix** | Changed CMD to `--artifacts-destination s3://...` ‚ÄĒ server proxies artifacts; client receives `mlflow-artifacts://` URI |

#### Issue 2 ‚ÄĒ MLflow Client Hanging at `set_experiment`

| | |
|---|---|
| **Symptom** | Python kernel hangs indefinitely at `mlflow.set_experiment()` |
| **Cause 1** | Tracking URI was `http://` ‚ÄĒ Cloudflare returns 308; MLflow client doesn't follow redirects |
| **Fix 1** | Changed tracking URI to `https://mlflow-geoai.stelarea.com` |
| **Cause 2** | MLflow 3.x telemetry fetch to `config.mlflow-telemetry.io` blocks in background thread |
| **Fix 2** | Set `os.environ["MLFLOW_DISABLE_TELEMETRY"] = "true"` **before** `import mlflow` |

#### Issue 3 ‚ÄĒ Client/Server Version Mismatch

| | |
|---|---|
| **Symptom** | Client (3.5.0) and server (2.14.3) API incompatibility; requests fail silently |
| **Fix** | Upgraded server to `mlflow==3.10.1`; updated client via `pip install mlflow==3.10.1` |

#### Issue 4 ‚ÄĒ Cross-Origin Request Blocked

| | |
|---|---|
| **Symptom** | `Blocked cross-origin request from https://mlflow-geoai.stelarea.com` in server logs; all mutating requests (POST, DELETE) return 403 |
| **Cause** | MLflow 3.x `CORSBlockingMiddleware` rejects cross-origin POST/DELETE/PUT when `--cors-allowed-origins` is not set |
| **Fix** | Added `--cors-allowed-origins "$MLFLOW_CORS_ALLOWED_ORIGINS"` to CMD; set `MLFLOW_CORS_ALLOWED_ORIGINS=https://mlflow-geoai.stelarea.com` in compose |

#### Issue 5 ‚ÄĒ Healthcheck `Rejected request with invalid Host header: localhost:8080`

| | |
|---|---|
| **Symptom** | Container healthcheck permanently UNHEALTHY; server logs `Rejected request with invalid Host header: localhost:8080` |
| **Cause** | `--allowed-hosts localhost` uses exact match in MLflow 3.x (`fnmatch` applies only when `*` present); `localhost:8080` does not match `localhost` |
| **Fix** | Changed to `localhost,localhost:*` |

#### Issue 6 ‚ÄĒ Caddy Not Forwarding Correct Host Header

| | |
|---|---|
| **Symptom** | Cross-origin errors persisted after CORS fix; server received `Host: localhost:8088` instead of `Host: mlflow-geoai.stelarea.com` |
| **Cause** | Default Caddy `reverse_proxy localhost:8088` forwards the internal host |
| **Fix** | User updated Caddyfile: `reverse_proxy localhost:8088 { header_up Host {host} }` |

### 4. `XAmzContentSHA256Mismatch` ‚ÄĒ Root Cause Identified

**File changed**: `mlflow-research/Dockerfile`
**File created**: `notebooks/boto3_s3_test.ipynb`

#### Background

The artifact upload chain fails with:
```
boto3.exceptions.S3UploadFailedError: ... XAmzContentSHA256Mismatch when calling the PutObject operation
```

#### Root Cause (Confirmed via Direct Testing)

**Not** a payload signing issue. The actual cause: **boto3 ‚Č•1.34 enables `request_checksum_calculation=when_supported` by default**, which adds CRC32 checksum headers (`x-amz-checksum-crc32`) to every `PutObject` request. MinIO rejects these headers with `XAmzContentSHA256Mismatch`.

The previous fix (`payload_signing_enabled = true` in `~/.aws/config`) addressed a different older issue (streaming SHA256 placeholder) and does NOT fix the boto3 ‚Č•1.34 checksum behavior.

#### Confirmed Fix

```python
# Python client
from botocore.config import Config
s3 = boto3.client("s3", ..., config=Config(
    request_checksum_calculation="when_required",
    response_checksum_validation="when_required",
))
```

```dockerfile
# Dockerfile (server-side boto3 for proxied uploads)
ENV AWS_REQUEST_CHECKSUM_CALCULATION=when_required \
    AWS_RESPONSE_CHECKSUM_VALIDATION=when_required
```

Verified: direct `upload_file` to `s3://research-geoai/mlflow/artifacts/test-boto3-direct/fixed.txt` succeeds with the fix; fails without it.
