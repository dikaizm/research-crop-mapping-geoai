### 4. `XAmzContentSHA256Mismatch` — Root Cause Identified

**File changed**: `mlflow-research/Dockerfile`
**File created**: `notebooks/boto3_s3_test.ipynb`

#### Background

The artifact upload chain fails with:
```
boto3.exceptions.S3UploadFailedError: ... XAmzContentSHA256Mismatch when calling the PutObject operation
```

#### Root Cause (Confirmed via Direct Testing)

**Not** a payload signing issue. The actual cause: **boto3 ≥1.34 enables `request_checksum_calculation=when_supported` by default**, which adds CRC32 checksum headers (`x-amz-checksum-crc32`) to every `PutObject` request. MinIO rejects these headers with `XAmzContentSHA256Mismatch`.

The previous fix (`payload_signing_enabled = true` in `~/.aws/config`) addressed a different older issue (streaming SHA256 placeholder) and does NOT fix the boto3 ≥1.34 checksum behavior.

#### Confirmed Fix

```python
# Python client
from botocore.config import Config
s3 = boto3.client("s3", ..., config=Config(
    request_checksum_calculation="when_required",
    response_checksum_validation="when_required",
))
```

```dockerfile
# Dockerfile (server-side boto3 for proxied uploads)
ENV AWS_REQUEST_CHECKSUM_CALCULATION=when_required \
    AWS_RESPONSE_CHECKSUM_VALIDATION=when_required
```

Verified: direct `upload_file` to `s3://research-geoai/mlflow/artifacts/test-boto3-direct/fixed.txt` succeeds with the fix; fails without it.

### 8. `notebooks/04_feature_analysis_v2.ipynb` — ASTFS-Inspired Per-Crop Band Selection

**File created**: `notebooks/04_feature_analysis_v2.ipynb`

Created a new feature analysis notebook implementing an ASTFS-inspired (Yin et al. 2020) per-crop band selection approach, replacing the global joint-score approach in v1.

#### Motivation

In v1, Stage 1 ranks bands by their global mean GSI across all crops, and Stage 2 trains a multiclass oracle (11 classes, metric = mIoU). This collapses per-crop spectral requirements into a single global ranking, potentially masking features that are highly discriminative for rare or spectrally similar crops. The ASTFS method instead computes a **per-crop SIglobal** and runs a **binary forward selection** (crop_i vs. rest) for each crop independently.

#### Key Differences vs v1

| Aspect | v1 (global) | v2 (ASTFS-inspired per-crop) |
|---|---|---|
| Stage 1 ranking | Global mean GSI + RF joint score | Per-crop SIglobal (`gsi_df[crop_id]`) |
| Stage 2 oracle | Multiclass U-Net (11 classes), metric = mIoU | Binary U-Net (2 classes), metric = IoU(class 1) |
| Output | Single K* band list (global) | Per-crop K*_i + union → Stage 3 Exp C |
| Candidates | One ranked list of 25 | One ranked list of 20 per crop (10 lists total) |

**Reference**: Yin et al. (2020). "Automatic Spectro-Temporal Feature Selection for Crop Mapping." Remote Sensing 12(1):162. https://doi.org/10.3390/rs12010162

#### Notebook Structure

| Section | Key implementation |
|---|---|
| Config | Same paths/params as v1; added `TOP_K_PER_CROP=20`, `CDL_CLASS_NAMES` dict |
| Data loading | Identical to v1 (S2 stack + CDL + pixel sampling at `SAMPLE_FRACTION=0.05`) |
| **Stage 1** | `gsi_df[crop_id].sort_values()` per crop → `candidates_per_crop[crop_id]`; SIglobal heatmap (crops × dates) |
| **Stage 2** | `RasterPatchDataset` upgraded with `remap_lut` and `target_class_id` params; `train_eval_binary()` trains binary U-Net; per-crop loop logged under `crop_{id}_` MLflow prefix |
| **Stage 2.3** | Per-crop results table {Crop \| Key Dates \| Key Bands \| K*_i \| Final IoU(class 1)}; union computation; per-crop IoU progression subplots |
| Stage 3 note | `stage3_exp_c` = union of per-crop selections → Exp C input for `05_train_segmentation.ipynb` |

#### Design Decisions

**`RasterPatchDataset` upgrades:**
- `remap_lut=None` — pass a custom binary LUT `{crop_id→1, else→0}` per crop oracle; defaults to global multiclass REMAP_LUT for backward compatibility
- `target_class_id=None` — if set, only include patches with ≥1 pixel of the target CDL class; ensures binary oracle trains on patches that have positive examples (critical for rare crops such as Grapes at 1.3%)

**`train_eval_binary()` vs `train_eval()` (v1):**
- `num_classes=2` (binary)
- `CrossEntropyLoss()` without `ignore_index=0` — in binary mode, class 0 (everything that is not the target crop) is informative; suppressing it would remove most of the supervision signal
- Metric: `compute_iou_class1()` = IoU of class 1 (the target crop) instead of mean IoU over all foreground classes

**Per-crop SIglobal rationale:**
`gsi_df = bs.calculate_gsi(df, CLASS_COL)` returns shape `(n_bands, n_classes)`. Column `gsi_df[crop_id]` is the SI between crop_id and all other classes, averaged over all pairings — exactly SIglobal(b) from ASTFS Eq. 2. Using the per-crop column instead of the global mean (`gsi_df.mean(axis=1)`) means that, e.g., a band that is excellent for distinguishing Rice from everything else but mediocre for Grapes will rank near the top for Rice but not for Grapes — a more targeted selection.

**Stage 3 union:**
The union of all per-crop selected bands (in first-occurrence order across crops) becomes `stage3_exp_c`. This is expected to produce a set of 15–30 bands that is both compact (vs. 44 for Exp B) and multi-crop optimal (vs. 4 for the v1 K*=4 under-selected result).

### 10. Methodology Analysis — Stage 2 Forward Selection: Order-Dependency and ASTFS Comparison

#### Background

A critical question was raised about the interpretability and validity of the Stage 2 CNN forward selection: **does the order in which bands are evaluated affect the final selected set?**

#### Finding 1 — Stage 2 is path-dependent (order matters)

Greedy forward selection is path-dependent. Each accept/reject decision is made given the currently accumulated set, so an early selection can block a later candidate from being chosen.

**Example:** If `B8_Jul` is evaluated before `B8_Mar`, one will be accepted and the other rejected as redundant. If the evaluation order is reversed, the opposite happens — the final selected bands differ even though they carry roughly equivalent information.

The root cause is that our Stage 2 evaluates candidates in a **fixed pre-determined order** (Stage 1 SIglobal rank), not the true greedy approach:

| Approach | Evaluation strategy | Complexity | Properties |
|---|---|---|---|
| True greedy forward selection | At each step, evaluate ALL remaining candidates; pick the one with the highest marginal gain | O(N²) | Locally optimal at each step |
| Our Stage 2 (ASTFS-inspired) | Evaluate candidates in fixed Stage 1 rank order; accept if IoU gain > δ | O(N) | Order-dependent; Stage 1 rank is a proxy for marginal gain from ∅ |

The key justification for the fixed-order approach is that Stage 1 SIglobal is a good proxy for what true greedy would select first (the best band from the empty set). Where the approximation weakens is in later steps, once several bands are already selected and inter-band redundancy matters.

#### Finding 2 — The original ASTFS paper does not acknowledge order-dependency

The ASTFS paper (Yin et al. 2020) lists four limitations in Section 4.2:
1. Other separability indicators could replace SI_ij
2. SI_global could be computed differently (min, max, weighted average)
3. Method is limited to single cropping
4. TOA instead of surface reflectance was used

**Order-dependency is not mentioned.** The paper does not discuss that the fixed SIglobal ranking makes the selection path-dependent, nor that this differs from true greedy search.

#### Finding 3 — Original ASTFS has no early stopping; our implementation does

Reading the ASTFS methodology carefully (Section 2.3.3) reveals a critical difference:

> *"...continue to repeat steps (4) through (5) **until all new features in the feature set are inputted** into RF classifier."*

The original ASTFS **exhausts all N candidates**. Every feature is tested regardless of how many consecutive rejections have occurred. Our implementation adds two modifications the original does not have:

| | Original ASTFS | Our implementation |
|---|---|---|
| Stopping criterion | Exhausts all candidates (all 77 features evaluated) | Stops after `S2_NO_IMPROVE=5` consecutive rejections |
| Acceptance threshold | Any improvement over previous best accuracy (δ=0) | Must exceed δ=0.005 IoU gain |
| Oracle | Random Forest classifier | Lightweight U-Net (ResNet-18) |

**Consequence:** Early stopping makes our version **more** order-sensitive than the original. In ASTFS, even if 10 mediocre bands appear consecutively, the algorithm continues and may still accept a good band at position 50. In our implementation, 5 consecutive rejections terminates the search entirely — a good band at position 50 never gets evaluated if the 5 bands before it are all rejected.

#### Implication for the thesis

Two distinct limitations should be acknowledged:

1. **Order-dependency (inherited from ASTFS, unacknowledged in the original paper):** The fixed Stage 1 ranking determines which band "gets credit" for shared spectral information. This is an approximation of true greedy search, justified by the O(N) vs O(N²) complexity trade-off (275 candidates × 15 epochs each → ~75,000 training runs for true greedy vs 275 for fixed-order).

2. **Early stopping amplifies order-sensitivity (our modification):** The `S2_NO_IMPROVE=5` criterion is a practical efficiency choice, but it couples the final selected set more strongly to the Stage 1 ordering than the original ASTFS design intended.

**Proposed mitigation — sensitivity analysis:** Run Stage 2 with randomly shuffled Stage 1 orderings (2–3 random seeds) and compare final K*, final IoU, and which bands appear consistently across orderings (stable selections) vs only in some orderings (order-sensitive selections). If final IoU is stable across orderings but specific band identity varies slightly, this supports the claim that selected *information content* is robust even if individual band identity is order-sensitive.

### 11.5 MLflow Nested Runs per Crop (Stage 2)

Stage 2 forward selection was refactored to use MLflow **nested runs**: one parent run for the stage as a whole, and one child run per crop class.

**Before:** A single flat run with metrics prefixed by crop ID (`crop_61_iou`, `crop_75_iou`) all sharing a global step counter — meaningless IoU curves.

**After:**
```
stage2v2_binary_fwd_{ts}           ← parent: hyperparams, per-crop summary
├── stage2v2_Rice_{ts}             ← child: iou_class1 vs step=band_rank
├── stage2v2_Sunflower_{ts}
├── ...
└── stage2v2_Plums_{ts}
```

Each child run logs:
- `iou_class1`, `gain`, `accepted`, `n_selected` at `step=band_rank` (clean IoU curve per crop)
- `final_iou`, `k_star` as summary metrics
- `selected_bands`, `stop_reason` as tags
- `crop_id`, `crop_name`, `n_candidates` as params

The parent mirrors `crop_{id}_final_iou` and `crop_{id}_k_star` so the summary is visible without drilling into children.

The `mlflow_run_id` in `stage2v2_per_crop_results.csv` now points to the specific crop child run.

### 11.7 Pipeline Log Upload to MLflow

After each full pipeline run, the `.log` file is uploaded as a MLflow artifact to a dedicated `cropmap_pipeline_runs` experiment. Log handlers are flushed before upload to ensure the file is complete (including the final summary). The pipeline run MLflow entry also records:

- Per-stage elapsed time and status
- Total wall time (minutes)
- `pipeline_status` tag (`ok` / `error`)

If MLflow is unreachable at upload time, the failure is logged as a warning and the pipeline still exits normally.

### 11.8 Auto-Shutdown After Experiments (`--shutdown`)

A `--shutdown` flag was added to `pipeline.py` to schedule a server shutdown 8 minutes after all stages complete. This runs after `_upload_log()` finishes, so all MLflow artifacts are safely uploaded before shutdown.

```bash
# Run full pipeline and auto-shutdown
./run.sh --stages all --shutdown

# Cancel if needed
sudo shutdown -c
```

The 8-minute window allows time to verify results in the MLflow UI before the server goes down.

### 11.9 Critical Bug Fix — `--data-dir` Override Scope

A critical bug was identified and fixed in both `feature_analysis.py` and `train_segmentation.py`: the `--data-dir` override was updating the config module object (`_cfg.S2_PROCESSED_DIR = ...`) but **not** the already-imported module-level names used by all stage functions. As a result, functions like `compute_class_weights()`, `load_data()`, and `run_experiment()` would still read from the original hardcoded paths when `--data-dir` was passed.

**Fix:** Both `main()` functions now use `global` declarations to reassign the module-level names directly:

```python
# train_segmentation.py
global S2_PROCESSED_DIR, CDL_BY_YEAR, MODELS_DIR, FIGURES_DIR
S2_PROCESSED_DIR = data_dir / "s2"
CDL_BY_YEAR      = {yr: data_dir / "cdl" / ... for yr in [...]}
MODELS_DIR       = data_dir / "models"
FIGURES_DIR      = data_dir / "figures"

# feature_analysis.py
global S2_PROCESSED_DIR, CDL_BY_YEAR, PROCESSED_DIR, FIGURES_DIR, \
       STAGE2_RESULTS_CSV, STAGE3_EXP_C_BANDS
```

`MODELS_DIR` and `FIGURES_DIR` were also added to the override (previously missing), so model checkpoints and figures are written to the custom data directory rather than the default project paths.
