| Plums | -0.370 | +0.680 | -0.271 | Unstable (small area, noise) |

### Decision: Stay with 2022 only for Stage 1+2

1. **Biological variability, not data scarcity** — Alfalfa r(23v24)=+0.174 and
   Plums r(23v24)=−0.271 show even 2023 and 2024 do not agree. Adding 2023 to
   Stage 1+2 would not improve band selection for these crops.
2. **Alfalfa** is a multi-cut perennial (3–5 cuts/year), cutting schedule shifts
   inter-annually. **Plums** covers only 2.1% of pixels — NDVI is dominated by
   mixed-pixel noise at downsampled resolution.
3. **Data leakage avoided** — 2023 is a Stage 3 training year.
4. **Option B fallback is the correct response** for K*=0 crops — the spectral/
   temporal ambiguity is intrinsic, not fixable by more reference data.
5. **Grapes is the only genuine case** where 2022 is anomalous (r(23v24)=+0.755).
   Handled by δ=0.001 + Option B re-run.

## 3. Stage Project — Per-Year Band Projection

**Problem**: `stage3_exp_c_bands.txt` stores 2022 dates (e.g. `B4_20220730`). In
`run_experiment()`, the same positional `band_indices` were applied to all years.
If 2023 has different dates or fewer files, index 157 in 2023 ≠ Jul-30 semantically.

**Solution**: a new `--stage project` command in `feature_analysis.py` maps each
selected MMDD to the nearest acquisition date in each year, then writes a per-year
projected JSON consumed by Stage 3.

### Files updated

| File | Change |
|---|---|
| `config.py` | Added `STAGE3_EXP_C_BANDS_PROJECTED` path |
| `feature_analysis.py` | Added `run_project()`, `--stage project` choice |
| `train_segmentation.py` | Added `build_exp_C_indices_projected()`, `_fetch_projected_from_mlflow()`, per-year index handling in `run_experiment()`, `--project-run-id` flag |

### `run_project()` logic

1. Reads `stage3_exp_c_bands.txt` → parses `(band, MMDD)` pairs
2. For each year in `TRAIN_YEARS + [TEST_YEAR]` (2022, 2023, 2024):
   - Scans `S2_PROCESSED_DIR/{year}/*_processed.tif` for available dates
   - For each MMDD: finds the nearest acquisition date (minimise DOY distance)
   - Warns if any match gap > 15 days
3. Writes `stage3_exp_c_bands_projected.json`:
   ```json
   {
     "2022": ["B4_20220730", "B8_20220730", ...],
     "2023": ["B4_20230801", "B8_20230801", ...],
     "2024": ["B4_20240730", "B8_20240730", ...]
   }
   ```
4. Logs artifact + per-year band counts to MLflow (`cropmap_feature_analysis_s2`,
   run name `stage2v2_project_YYYYMMDD-HHMMSS`)

### `build_exp_C_indices_projected()` in `train_segmentation.py`

- Loads projected JSON (local or from MLflow via `--project-run-id`)
- For each year: builds a year-specific `band_name → flat_channel_index` map
  from that year's sorted S2 files
- Returns `dict {yr: (idx_list, names_list)}`
- Falls back to `None` → caller uses single-ref-year behaviour (Exp A/B unaffected)

### `run_experiment()` changes

- Accepts `band_indices` as `list[int]` (uniform, Exp A/B) or
  `dict{yr: (idx, names)}` (per-year, Exp C projected)
- Internal `_yr_idx(yr)` helper resolves the correct indices per year
- Both train years and test year use their own year-specific indices

### New CLI flags

```bash
--project-run-id   MLflow run ID of stage2v2_project run (auto-search if omitted)
--stage2-run-id    MLflow run ID of stage2v2_binary_fwd run (auto-search if omitted)
```

### 5.2 GDAL Tile Decode Errors (`TIFFReadEncodedTile`, `LZWDecode`, `ZIPDecode`)

**Problem**: Stage 3 training crashed with DataLoader worker exceptions:

```
TIFFReadEncodedTile() failed.
LZWDecode: Not enough data at scanline 3584
ZIPDecode: Decoding error at scanline 5120
```

**Root cause — Stage 2 vs Stage 3 difference**:
- Stage 2 pre-loads all patches into `TensorDataset` in RAM → GDAL errors silently
  return zeros, training continues
- Stage 3 `RasterPatchDataset.__getitem__` reads lazily per-tile in DataLoader workers
  → GDAL errors raise Python exceptions → worker crashes

**Root cause — corrupt tiles**: Some raw S2 TIFs exported from GEE have a small number
of tiles with corrupted LZW streams. These were written into `*_processed.tif` output
during processing.

**Fix 1 — `geoai/geoai/train.py`**: Added try/except around `src.read(window=win)` in
`RasterPatchDataset.__getitem__`. On GDAL error, fills the tile with nodata (zeros)
instead of raising:

```python
try:
    arr = src.read(window=win).astype(np.float32)
except Exception:
    arr = np.full((src.count, ps, ps), self.nodata, dtype=np.float32)
```

GDAL still logs the error as `INFO` — visible in logs but non-fatal.

**Fix 2 — `crop_mapping_pipeline/stages/process_data.py`**: Changed compression from
LZW to DEFLATE to avoid LZW + float32 predictor incompatibility across GDAL versions:

```python
# Before
profile.update(dtype="float32", nodata=S2_NODATA, compress="lzw", predictor=2)
# After
profile.update(dtype="float32", nodata=S2_NODATA, compress="deflate", predictor=2)
```

Newly processed files will use DEFLATE. Already-processed files on the server retain
LZW but are handled gracefully by Fix 1.

**Pre-flight validation** added to `train_segmentation.py` `main()`: reads the first
256×256 tile of every S2 file before training begins. Lists all corrupt files and
raises `RuntimeError` if any are found, rather than failing mid-training.

### 5.4 BatchNorm Crash on Last Batch (`drop_last=True`)

**Problem**: Training crashed with:

```
ValueError: Expected more than 1 value per channel when training,
got input size torch.Size([1, 256, 1, 1])
```

**Root cause**: DeepLabV3+'s ASPP module contains a global average pooling branch that
squeezes spatial dimensions to 1×1. BatchNorm in training mode requires ≥2 values per
channel. When the last batch has exactly 1 sample and spatial=1×1, it has 1 value →
crash.

**Fix — `geoai/geoai/train.py`**: Added `drop_last=True` to the training DataLoader:

```python
train_loader = DataLoader(
    train_dataset,
    batch_size=batch_size,
    shuffle=True,
    num_workers=num_workers,
    pin_memory=True,
    drop_last=True,   # ← added: discard final batch if size < batch_size
)
```

The last incomplete batch (e.g. 1 sample from 1001 total with batch_size=8) is silently
discarded each epoch, preventing the BatchNorm crash.

## 9. Feature Analysis v2 — Date × Band Structured Selection

### 9.1 Motivation

The original Stage 2 selects arbitrary `(date, band)` channel pairs independently,
producing a flat list of channel indices that is hard to interpret agronomically.

v2 produces a **structured per-crop output**:
```json
{"Rice": {"dates": ["20220715", "20220730"], "bands": ["B8", "B4", "B11"]}, ...}
```
The final Exp C v2 input = `union_dates × union_bands` (cartesian product).

### 9.2 New Files

| File | Description |
|---|---|
| `stages/feature_analysis_v2.py` | New — Stage 1v3 + Stage 2v2 pipeline |

### 9.3 Algorithm

**Stage 1v3 — Date + Band Ranking (CPU)**
- Call `bs.calculate_gsi()` once → 275 × 10 SI score matrix
- Aggregate by date: mean SI across 11 bands per date per crop → top `TOP_DATES_PER_CROP=10`
- Aggregate by band: mean SI across 25 dates per band per crop → top `TOP_BANDS_PER_CROP=9`
- Output: `stage1v3_candidates.json`

**Stage 2v2-A — Date Forward Selection (GPU)**
- Per crop: iterate ranked date candidates
- Oracle input = `selected_dates × all 9 VEGE_BANDS` (full spectral context)
- Accept date if `IoU > best + S2_DATE_DELTA=0.010`
- Stop after `S2_DATE_NO_IMPROVE=4` rejections or `S2_MAX_DATES=8`

**Stage 2v2-B — Band Forward Selection (GPU, fixed dates)**
- Per crop: iterate ranked band candidates with dates fixed from Phase A
- Oracle input = `K*_dates × selected_bands`
- Accept band if `IoU > best + S2_BAND_DELTA=0.005`
- Stop after `S2_BAND_NO_IMPROVE=4` rejections or `S2_MAX_BANDS_V2=9`

Both phases: fallback to top-1 if nothing accepted.

## 10. Exp B — NaN Loss Fix

### 10.1 Root Cause

Exp B uses 4 phenological dates (Jan, Mar, Jul, Nov). Winter acquisitions (Jan, Mar, Nov)
have cloud-masked pixels stored as `NaN` or `inf` rather than the exact `-9999` sentinel.
These slip past the `arr[arr == self.nodata] = 0.0` check.

When a NaN/inf pixel enters the forward pass, **BatchNorm updates its running statistics
with NaN** (in `model.train()` mode), permanently corrupting the model. All subsequent
forward passes produce NaN outputs → NaN loss → mIoU = 0 forever.

Pattern observed:
- Epoch 1: loss=1.8888, mIoU=0.2666 (first epoch fine — NaN batch hits mid-epoch,
  corrupts BatchNorm for remaining batches of epoch 1 but average still finite)
- Epoch 2+: loss=nan, mIoU=0.0000 (all batches corrupt)

### 10.2 Fix 1 — Dataset (`geoai/geoai/train.py`)

Added `arr[~np.isfinite(arr)] = 0.0` in `RasterPatchDataset.__getitem__` after the
nodata assignment, to catch NaN/inf values that aren't the exact nodata sentinel:

```python
arr[arr == self.nodata] = 0.0
arr[~np.isfinite(arr)] = 0.0   # catch residual NaN/inf from cloud masking
```

### 10.3 Fix 2 — Training loop (`geoai/geoai/train.py`)

Added `torch.nan_to_num` **before** the forward pass in `train_semantic_one_epoch`,
preventing any remaining NaN/inf from corrupting BatchNorm running stats:

```python
images = torch.nan_to_num(images, nan=0.0, posinf=1.0, neginf=0.0)
outputs = model(images)
```

Added loss guard as backup (skips backward but not forward — secondary defence):
```python
if not torch.isfinite(loss):
    optimizer.zero_grad()
    continue
```

### 10.4 Fix 3 — Validation (`train_segmentation.py`)

Same `nan_to_num` guard added to `validate_one_epoch` for consistency:
```python
imgs = torch.nan_to_num(imgs, nan=0.0, posinf=1.0, neginf=0.0)
```

## 11. Experiment D — Stage 1 GSI Ablation (no CNN Forward Selection)

### 11.1 Motivation

The experiment matrix now includes a fourth configuration to serve as an ablation
of the CNN forward selection step in Stage 2:

| Config  | Input                             | Channels | Purpose                        |
|---------|-----------------------------------|----------|--------------------------------|
| Exp A   | Single-date (Jul 30)              | 9        | Conventional baseline          |
| Exp B   | 4 phenological dates              | 36       | Multi-temporal naive           |
| Exp C   | Stage 2 CNN forward-selection     | K*       | Proposed method                |
| **Exp D** | **Stage 1 GSI top-K (no Stage 2)** | **K\*** | **Ablation: no CNN validation** |

Exp D answers: *how much does the CNN forward selection (Stage 2) actually contribute
over simply taking the top-ranked bands from the GSI metric?*  If Exp C ≫ Exp D,
it demonstrates that the CNN oracle meaningfully prunes irrelevant channels.
If Exp C ≈ Exp D, the GSI ranking alone is sufficient and Stage 2 adds complexity
without benefit.

### 11.2 Design

Exp D reads from `stage3_exp_d.json`, produced automatically by `feature_analysis_v2.py`
at the end of Stage 1v3.

**Feature construction:**
1. For each crop, Stage 1v3 produces a ranked date list and a ranked band list
   (both by mean SI_global).
2. Union of `TOP_DATES_PER_CROP` dates across all 10 crops → `union_dates`
3. Union of `TOP_BANDS_PER_CROP` bands across all 10 crops → `union_bands`
4. Input = `union_dates × union_bands` channel combinations (same format as Exp C v2)

No CNN oracle; no forward selection. Pure GSI-ranked feature set.

### 11.3 Implementation

**`config.py`**
```python
STAGE3_EXP_D_JSON  = PROCESSED_DIR / "stage3_exp_d.json"
STAGE3_EXP_D_BANDS = PROCESSED_DIR / "stage3_exp_d_bands.txt"
```
