```

Stacking cell (new)
  └─ Stack 3 S2 files → (27, H, W) multi-temporal array
  └─ Named bands: B2_20220101 ... B12_20220131
  └─ Filter valid pixels (∈ KEEP_CLASSES, all-finite)
  └─ Sample SAMPLE_FRACTION → df for Stage 1

  └─ MLflow run: stage1_gsi_rf_{timestamp}

  └─ Logs: per-step miou/gain/accepted metrics, final_miou, selected bands
  └─ Artifacts: stage2_forward_selection_history.csv
```

## Section 5 — Feature Selection Pipeline & Experiment Design Clarification

### 5.1 Stage 1 → Stage 2 Relationship

Stage 1 and Stage 2 are directly linked and interdependent — Stage 2 cannot function without Stage 1.

| Stage | Role | Output |
|---|---|---|
| **Stage 1** (GSI + RF joint score) | Scores and **ranks** all 648 band-date features by discriminative power | Ordered feature list: `[B8_Jul30, B5_Mar17, B11_Jul30, ...]` |
| **Stage 2** (CNN forward selection) | Evaluates features **in Stage 1 order** sequentially to find the optimal cut-off | Optimal subset of $K^*$ bands |

**Stage 2 is not a pure greedy search.** Pure forward selection (no pre-ranking) would require testing all remaining features at each step → $O(N^2) \approx 210{,}000$ model evaluations for $N = 648$.

By using the Stage 1 ranking as a fixed addition order, Stage 2 only evaluates $O(N) = 648$ sequential subsets:

```
Step 1:  [B8_Jul30]                   → mIoU = 0.42
Step 2:  [B8_Jul30, B5_Mar17]         → mIoU = 0.51
Step 3:  [..., B11_Jul30]             → mIoU = 0.58
...
Step K*: accuracy plateaus → stop     → K* = optimal cut-off
```

**Stage 1 determines the ORDER of addition; Stage 2 determines HOW MANY to keep (K\*).**

### 5.2 Revised Experiment Design

The full model validation (Stage 3) uses three input configurations to answer two research questions progressively:

| Config | Input | Channels | Purpose |
|---|---|---|---|
| **Exp A**: Single-date baseline | 9 bands × 1 date (Jul 30, peak season) | 9 | No temporal information |
| **Exp B**: Multi-temporal naive baseline | 9 bands × 4 phenological dates (Jan, Mar, Jul, Nov) | 36 | Temporal without selection |
| **Exp C**: Multi-temporal selected (proposed) | $K^*$ bands from Stage 2 output | $K^* < 36$ | Temporal + optimal selection |

- **A → B** measures: does temporal information improve over single-date?
- **B → C** measures: does the proposed selection find a more compact subset without accuracy loss?

Each configuration × 2 architectures (DeepLabV3+CBAM, SegFormer) = **6 total runs**.

### 5.3 Why "All Dates, All Bands" is Not a Valid Baseline

The naive "full multi-temporal" input would be 72 dates × 9 bands = **648 channels**, which is computationally infeasible and defeats the purpose of feature selection as a research contribution.

The seasonal 4-date baseline (Exp B, 36 channels) serves as the principled "temporal without selection" baseline — it covers the full phenological cycle with a fixed, calendar-based date choice (no algorithm), making it a fair comparison point for the proposed method.
