## Summary of Changes

### 1. `process_data.py` — New Processing Stage Script

**File created**: `crop_mapping_pipeline/stages/process_data.py`

Created a production-ready processing script ported from `notebooks/02_image_processing.ipynb`. Handles year-by-year S2 processing and CDL reprojection with automatic upload to Google Drive after each year.

#### Key Features

| Feature | Description |
|---|---|
| Year-by-year processing | `--years 2022 2023 2024` or individually; minimises VPS disk usage |
| S2 NoData assignment | Invalid/negative/NaN pixels → `-9999`, cast to `float32` |
| CDL reprojection | `rasterio.warp.reproject()` EPSG:5070→4326, clip+resample to S2 grid in one call |
| CDL class filtering | Applies `REMAP_LUT` — keeps `KEEP_CLASSES`, maps to sequential 1–10, rest → 0 |
| GDrive upload | Uploads processed files per year after processing; OAuth 2.0 credentials |
| Auto-shutdown | `--shutdown` flag calls RunPod API (`podStop` mutation) when done |
| `--auth` flag | Generates OAuth token locally via `InstalledAppFlow` (run before deploying to VPS) |

#### Processing Order (NoData before resampling)

```
Raw S2 TIFs → assign NoData (invalid/neg/NaN → -9999, float32) → save *_processed.tif
Raw CDL (EPSG:5070, 30m) → reproject+clip+resample to S2 grid (single rasterio call) → filter classes
Verify: same CRS, bounds, dimensions
Upload processed files to GDrive → (optionally) delete raw to free disk
```

NoData is assigned **before** resampling to prevent bilinear/nearest interpolation from blending valid pixels with uninitialized border values.

#### CLI Usage

```bash
# Generate OAuth token locally (run once before deploying)
python process_data.py --auth

# Process 2022 only, then upload and delete raw
python process_data.py --years 2022 --delete

# Process all years and auto-shutdown VPS when done
python process_data.py --years 2022 2023 2024 --shutdown
```

### 2. OAuth 2.0 for Google Drive Uploads

**Problem**: Service account credentials (`gdrive_service_account.json`) cannot upload to a personal Google Drive — service accounts have no quota on personal Drive and return `403 storageQuotaExceeded`.

**Solution**: Switched to OAuth 2.0 with `InstalledAppFlow` (Desktop app client type). Token is generated locally and transferred to the VPS.

#### Setup Flow

1. Create OAuth 2.0 client ID of type **Desktop app** in Google Cloud Console
2. Download `client_secret_*.json` → `crop_mapping_pipeline/ssh/`
3. Run `python process_data.py --auth` locally → opens browser → saves `gdrive_token.pickle`
4. Upload token to VPS via `0x0.st` (scp/sftp not available through RunPod SSH gateway)

```bash
# Upload token to temporary host
curl -F "file=@crop_mapping_pipeline/ssh/gdrive_token.pickle" https://0x0.st

# On VPS: download token
wget -O crop_mapping_pipeline/ssh/gdrive_token.pickle <URL_FROM_0x0.st>
```

#### Token Auto-Refresh

`_build_drive_service()` loads the pickle, checks expiry, and calls `credentials.refresh(Request())` if expired — no manual token rotation needed.

#### Errors Encountered and Fixed

| Error | Cause | Fix |
|---|---|---|
| `403 storageQuotaExceeded` | Service account cannot upload to personal Drive | Switch to OAuth 2.0 |
| `Error 400 redirect_uri_mismatch` | OAuth client was "Web application" type | Recreate as "Desktop app" type |
| `Error 403 access_denied` | App in testing mode, user email not whitelisted | Add `dikamaah@gmail.com` as test user in OAuth consent screen |
| `subsystem request failed` | RunPod SSH gateway doesn't support SFTP | Use `0x0.st` for file transfer |
| `SSH client doesn't support PTY` | RunPod base64 pipe approach also fails | Upload to `0x0.st`, `wget` on server |

### 3. RunPod API Auto-Shutdown

**Problem**: `sudo shutdown -h now` fails in RunPod containers — `System has not been booted with systemd as init system (PID 1)`.

**Solution**: RunPod GraphQL API `podStop` mutation called via `requests` library.

```python
def _schedule_shutdown():
    pod_id  = os.getenv("RUNPOD_POD_ID")
    api_key = os.getenv("RUNPOD_API_KEY")
    if pod_id and api_key:
        # RunPod GraphQL API
        requests.post(
            "https://api.runpod.io/graphql",
            headers={"Authorization": f"Bearer {api_key}"},
            json={"query": f'mutation {{ podStop(input: {{podId: "{pod_id}"}}) {{ id }} }}'},
        )
    else:
        subprocess.run(["sudo", "shutdown", "-h", "+8"])
```

#### `.env` File for Secrets

```bash
# crop_mapping_pipeline/.env
RUNPOD_API_KEY=<your-key>
RUNPOD_POD_ID=<your-pod-id>
```

Loaded via `python-dotenv` (`load_dotenv(_ROOT / ".env")`) in all stage scripts.

### 8. `feature_analysis.py` — Stage 2 RAM Optimisation

**File updated**: `crop_mapping_pipeline/stages/feature_analysis.py`

`load_data()` added a `stage` parameter to skip the full raster stack when only band names are needed.

#### Problem

Stage 2 called `load_data()` which stacks all 25 S2 TIFs (~29 GB RAM). On an RTX 2000 Ada (31 GB RAM), this left only 2 GB for PyTorch — too tight.

#### Solution

```python
def load_data(s2_year="2022", stage=1):
    """
    stage=1: full stack for GSI pixel sampling (~29 GB)
    stage=2: derive band names from filenames only (~2 GB)
    """
    tif_files = sorted(glob(f"{S2_PROCESSED_DIR}/{s2_year}/*_processed.tif"))
    band_names = _derive_band_names_from_files(tif_files)  # no rasterio read needed

    if stage == 2:
        return None, None, band_names, tif_files  # skip stacking

    # stage=1: full stack for pixel sampling
    s2_stack, cdl_arr = _stack_all_rasters(tif_files)
    return s2_stack, cdl_arr, band_names, tif_files
```

Band names (e.g. `B8_20220317`) are fully derivable from filenames without opening TIF data. Stage 2 only requires filenames to build `RasterPatchDataset` — no global stack needed.

**RAM impact**: 29 GB → ~2 GB for Stage 2. RTX 2000 Ada (31 GB RAM) is now sufficient.

#### 13.5 GPU Utilisation Fix — `_preload_patches()`

**Problem**: GPU utilisation was 0% despite `Device: cuda (...)` in the log.

**Root cause**: `RasterPatchDataset` opens 25 rasterio file handles in `__init__` (comment on line 113: *"num_workers=0 required — rasterio handles cannot be pickled"*). This forced `DataLoader(num_workers=0)` — every batch blocked on CPU I/O: open 25 TIF files → read all 275 channels → subset to 1–20 bands → GPU gets batch. CPU I/O and GPU compute alternated serially, leaving the GPU idle.

**Fix**: Added `_preload_patches(dataset)` — loads all patches from `RasterPatchDataset` into a `TensorDataset` (fully picklable) in a single pass before training:

```python
tensor_ds = _preload_patches(dataset)   # ~seconds, paid once per band trial
# Now safe to use num_workers + pin_memory
train_dl = DataLoader(tensor_ds, batch_size=S2_BATCH_SIZE, shuffle=True,
                      num_workers=4, pin_memory=True, persistent_workers=True)
```

| | Before | After |
|---|---|---|
| DataLoader workers | 0 (forced) | 4 |
| `pin_memory` | False | True (CUDA only) |
| `persistent_workers` | — | True |
| GPU utilisation | ~0% | Active |
| Per-epoch data source | rasterio reads (I/O bound) | RAM tensors |
| Pre-load cost | — | ~seconds once per trial |

`TensorDataset` is imported from `torch.utils.data` (no new dependency). Pre-load memory footprint: `640 patches × 128×128 × 1–20 bands × 4 bytes` ≈ 42 MB–840 MB, well within 31 GB RAM.
