## 1. Stage 1v3 — NaN Bug Fix (SI Computation)

### 1.1 Symptom

Running Stage 1v3 produced a candidates JSON where every crop had identical date
and band selections (always the first 10 calendar dates, always B1–B8A in alphabetical
order). The GSI heatmap reported `global_vmax = nan`.

### 1.2 Root Cause

S2 processed TIFs store NoData as `-9999.0` (set correctly in `process_data.py`).
When loaded in `feature_analysis_v2.py`, NoData is converted in-memory:

```python
arr[arr == S2_NODATA] = np.nan
```

Cloud cover and shadow pixels cause some channels to be NaN for some pixels.
The per-crop SI computation then used non-NaN-safe statistics:

```python
mean_c = X_c.mean(axis=0)   # returns NaN if any pixel in the column is NaN
std_c  = X_c.std(axis=0)
mean_r = X_r.mean(axis=0)
si     = np.abs(mean_c - mean_r) / (std_c + 1e-9)
```

With all-NaN SI values, Python's `sorted()` falls back to insertion order — every
crop received identical results.  `process_data.py` was not the issue; the bug was
entirely downstream in the SI computation.

### 1.3 Fix — `feature_analysis_v2.py`

**1. NaN-safe SI statistics:**
```python
# Before
mean_c = X_c.mean(axis=0)
std_c  = X_c.std(axis=0)
mean_r = X_r.mean(axis=0)

# After
mean_c = np.nanmean(X_c, axis=0)
std_c  = np.nanstd(X_c, axis=0)
mean_r = np.nanmean(X_r, axis=0)
```

**2. NaN-safe log stats:**
```python
# Before
log.info(f"SI range: min={_v.min():.4f}  p95={np.percentile(_v,95):.4f} ...")

# After
log.info(f"SI range: min={np.nanmin(_v):.4f}  p95={np.nanpercentile(_v,95):.4f} ...")
```

**3. NaN-safe heatmap vmax:**
```python
# Before
global_vmax = float(np.percentile(all_vals, 95)) if all_vals else 1.0

# After
global_vmax = float(np.nanpercentile(all_vals, 95)) if all_vals else 1.0
```

**4. NaN diagnostic log added after pixel sampling:**
```python
_nan_px = np.isnan(_X_sample).any(axis=1).sum()
_nan_ch = np.isnan(_X_sample).any(axis=0).sum()
log.info(f"NaN in sample: {_nan_px:,} pixels ({100*_nan_px/len(df):.1f}%) "
         f"have ≥1 NaN channel;  {_nan_ch}/{n_channels} channels have ≥1 NaN pixel")
```

### 1.4 Verified Result

After the fix, Stage 1v3 produced differentiated per-crop selections reflecting actual
phenological signatures:

- Rice (3): top dates centred on Jul–Sep (peak growing season)
- Winter Wheat (24): top dates include Dec–Jan (winter growing period)
- Sunflower (6): top dates in Oct–Nov (harvest period)
- Almonds / Walnuts: top dates in spring (bloom) and autumn (harvest)

Each crop also selects different top bands, confirming the SI computation is now
working correctly.

## 2. Training — Best-Val Per-Class IoU Logged to MLflow

### 2.1 Change

`train_segmentation.py` previously logged `best_val_miou` (scalar) but not the
per-class breakdown at the best epoch.  Test per-class IoU was already logged
as `test_iou_{class_name}` metrics.

Three changes to `run_experiment()`:

```python
# 1. Initialise alongside best_miou
best_miou              = 0.0
best_val_per_class_iou = {}

# 2. Update when new best checkpoint is saved
if val_m["miou"] > best_miou + EARLY_STOP_DELTA:
    best_miou              = val_m["miou"]
    best_val_per_class_iou = val_m["per_class_iou"]   # ← new
    ...

# 3. Log after training ends (same pattern as test per-class)
for cls_id, iou in best_val_per_class_iou.items():
    if not np.isnan(iou):
        cdl_id = KEEP_CLASSES[cls_id - 1]
        name   = CDL_CLASS_NAMES.get(cdl_id, f"cls{cls_id}")
        mlflow.log_metric(
            f"best_val_iou_{name.lower().replace('/', '_').replace(' ', '_')}",
            iou,
        )
```

MLflow now shows `best_val_iou_rice`, `best_val_iou_almonds`, etc. alongside
`test_iou_rice`, `test_iou_almonds` — enabling direct val/test comparison per class
in the MLflow UI.

## 4. Discussion — Exp D Design Issue

The current Exp D output (Stage 1 GSI per-crop union) is not a meaningful ablation.
Taking the union of 10 crops × top-10 dates produces **24/25 dates** selected — almost
the full dataset.  This defeats the purpose of feature selection and makes Exp D ≈
"all data", indistinguishable from a full multi-temporal baseline.

**Proposed fix (not yet implemented):** Change `_save_exp_d_bands()` in
`feature_analysis_v2.py` to use **global top-K** ranking (mean SI across all crops)
rather than per-crop union.  Example: global top-8 dates × global top-5 bands = 40
channels — a compact, fair ablation that answers: *is global GSI ranking sufficient,
or does per-crop CNN personalization (Stage 2) add value?*

### 4.1 Discussion — Setup B (`Fallow/Idle` + Rare Crops as `Other Crops`)

Question raised: should `Fallow/Idle Cropland (CDL 61)` remain background, or be
merged into an `Other Crops` class together with rare crop classes below a coverage
threshold?

Current position:

- For the main thesis setup, keeping `61` as background is still the cleaner and more
  defensible decision.
- If `Other Crops` is introduced, it should be treated as an explicit alternative
  experimental setup rather than replacing the current label design.

Reasoning:

- `Fallow/Idle` is not a stable crop phenotype; it can look like bare soil, weeds,
  residue, or transitional fields depending on timing.
- A merged `Other Crops` class is highly heterogeneous and may become harder for the
  model to learn than either background or well-defined crop classes.
- For the thesis objective, the current formulation remains stronger:
  **10 target crops + background**, with non-target / rare / fallow classes outside
  scope.

Implementation discussion for Setup B:

- There are two distinct variants:
  - `Setup B1`: only Stage 3 uses an `Other Crops` class.
  - `Setup B2`: Stage 1, Stage 2, and Stage 3 all treat `Other Crops` as a target.
- `Setup B1` is much easier and is the recommended ablation path.
- `Setup B2` is a method change, not just a label change, because Stage 1 and Stage 2
  currently assume each target corresponds to a single CDL ID.

Pipeline/code impact identified:

1. Label regeneration is required first.
   The current processed CDL rasters already discard `61` and all classes outside
   `KEEP_CLASSES`, so Setup B cannot be tested on the existing filtered labels.

2. Core pipeline files that would need modification:
   - `crop_mapping_pipeline/config.py`
   - `crop_mapping_pipeline/utils/label.py`
   - `crop_mapping_pipeline/stages/process_data.py`
   - `crop_mapping_pipeline/stages/train_segmentation.py`

3. Notebook paths that would need matching changes:
   - `notebooks/05_train_segmentation.ipynb`
   - `notebooks/04_feature_analysis_v2.ipynb` (only if pursuing full `Setup B2`)

4. Stage 1 / Stage 2 impact:
   - Current feature-selection code loops over `KEEP_CLASSES` where each target is a
     single CDL ID.
   - Binary masks are built using exact equality (`y == crop_id`) and one-vs-rest
     logic over `KEEP_CLASSES`.
   - A synthetic grouped class such as `Other Crops` would require a new target-group
     representation instead of the current single-ID assumption.

Recommended research path:

- Keep the main thesis experiments unchanged: `10 crops + background`.
- If testing Setup B, implement it first as a Stage 3-only ablation (`Setup B1`) so
  the comparison isolates taxonomy effects without simultaneously changing the Stage 1
  and Stage 2 selection procedure.
- Only extend Setup B into Stage 1/2 if the Stage 3-only ablation shows clear value.

### 4.2 Discussion — What Each Stage 1 Version Does

There are currently three practical Stage 1 variants across the notebooks and
pipeline:

**Stage 1 v1**

- Location: `notebooks/04_feature_analysis_v1.ipynb`
- Method: global filter preselection using **GSI + RF importance → joint score**
- Output: one shared **global top-K** channel list for all crops
- Role: early baseline / cheap global filter

**Stage 1 v2**

- Location:
  - `notebooks/04_feature_analysis_v2.ipynb`
  - `crop_mapping_pipeline/stages/feature_analysis.py`
- Method: **per-crop SIglobal** ranking over full channel tokens such as
  `B4_20220730`
- Output: `stage1v2_candidates.json` with `candidates_per_crop`
- Role: legacy per-crop Stage 1 used by the original Stage 2 forward-selection
  pipeline

**Stage 1 v3**

- Location:
  - `crop_mapping_pipeline/stages/feature_analysis_v2.py`
  - `crop_mapping_pipeline/stages/selections/feature_analysis_v2/stage1/v3.py`
- Method:
  - per-crop one-vs-rest SIglobal
  - rank **dates per crop** first
  - rank **bands per crop within selected dates**
- Output:
  - `stage1v3_candidates.json`
  - Stage 3 ablation artifact for Exp D
- Role: current active Stage 1 for the newer two-phase `feature_analysis_v2`
  workflow

Practical interpretation:

- `v1` = global top-K filter
- `v2` = per-crop top-K **channel** ranking
- `v3` = per-crop top-K **date + band** ranking

Current recommendation:

- Treat **v3** as the main active Stage 1 for the thesis pipeline.
- Treat **v2** as the legacy baseline.
- Treat **v1** as an earlier global-filter baseline rather than the main method.

### 4.3 Discussion — Stage 1 v2 Direct-to-Stage-3 Channel Count

Question raised: if Stage 1 v2 provides `20` candidate channels for each of `10`
crops, should the Stage 3 input be capped because `10 × 20 = 200` entries sounds
too large?

Check performed on:

- `crop_mapping_pipeline/stages/stage1v2_candidates.json`

Result:

- Total crop-wise candidate entries: `200`
- Unique union across all crops: **62 channels**
- Unique dates represented: `15`
- Unique spectral bands represented: `11`

Interpretation:

- There is already substantial overlap across crops.
- The effective Stage 3 input is therefore **62 unique channels**, not 200.
- Since 62 channels is already within a reasonable experimental range, no additional
  capping heuristic is needed at this stage.

Current decision:

- Use the **ordered unique union** of Stage 1 v2 candidates directly for the Stage 3
  legacy baseline.
- Do **not** apply an additional cap (e.g. top-40 / top-50) unless later experiments
  show a strong practical need.

## 7. Decision and What To Do Next (Band Selection Focus)

### Decision

1. **Do not promote Dv2 as the new main method yet.**
2. **Keep Exp A segformer as current benchmark to beat** (`test_mIoU = 0.1937`).
3. **Use DeepLab signal as supporting evidence** that selection can help, but enforce fair ablations first.

### Immediate Action Plan (priority order)

1. **Run a clean loss-controlled ablation** (highest priority):
  - Re-run Dv2 with `WeightedCrossEntropy` (same as A/B/C), no other changes.
  - Optionally run A/B/C with `PhenologyAwareLoss` to isolate loss effect separately.
2. **Constrain channel budget for Stage 3 input**:
  - Test compact sets: `12`, `20`, `36` channels only (avoid immediate 62ch union for SegFormer).
  - Keep architecture-specific selection if needed (SegFormer and DeepLab may prefer different subsets).
3. **Protect key perennial crops in scoring objective**:
  - Add class-priority weighting in Stage 2 acceptance criterion (Almonds, Walnuts, Grapes, Rice).
  - Reject candidates that increase mean IoU but degrade the priority set beyond tolerance.
4. **Penalize temporal redundancy in selector**:
  - Add diversity constraint (minimum date spacing or similarity penalty) to avoid over-concentrated/duplicated phenology windows.
5. **Use 2024-aware validation for selector robustness**:
  - Keep 2022 for fast search, but score candidate sets with lightweight cross-year check (e.g., small holdout from 2023 or proxy split) before finalizing Stage 3 channels.
